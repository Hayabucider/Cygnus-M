# Cygnus-M Typing Trainer 仕様書

> 最終更新: 2026-06-23  
> 対象ファイル: `docs/cygnus_m_typing_trainer.html`  
> 正典ファイル: `config/Cygnus.keymap`

---

## 目的

```
このTyping Trainer は、Cygnus-M の独自キーマップを身体に覚えさせるための練習ツールである。
正典はあくまで config/Cygnus.keymap であり、Trainer は正典ではない。
Trainer はデザイン成果物ではなく、キーマップ習得のための補助ツールである。
```

通常のタイピング練習サイトでは、独自配列・レイヤー操作・Tap/Hold・親指キーの練習ができない。
このツールはCygnus-M専用として、実機と連動した正誤判定・レイヤーハイライトを提供する。

---

## 正典の定義とファイルの関係

このプロジェクトのファイルは以下の順で派生している。

```
config/Cygnus.keymap          ← 唯一の正典
        ↓ キー配置・レイヤー定義を継承
docs/Cygnus-M-viewer.html       ← 正典の可視化ツール（第1世代派生）
        ↓ キー配置・レイヤー定義・UIを継承
docs/cygnus_m_typing_trainer.html ← 練習ツール（第2世代派生）
```

**重要な原則:**

- Trainer のキー配置・レイヤー構成は `config/Cygnus.keymap` を正典とする
- Viewer と Trainer の間に矛盾が生じた場合、keymap を正として両方を修正する
- Trainer だけを根拠に keymap や Viewer を修正しない

## 開発経緯

### フェーズ1: キーマップ構築（Claude / Gemini / ChatGPT）

`config/Cygnus.keymap` をメインの開発対象として、ZMK firmware 向けの11レイヤーキーマップを構築した。複数AIによる設計レビュー・修正・検証を経て確定。

### フェーズ2: Viewer 開発（Claude）

keymap の内容をブラウザ上で視覚確認するためのHTMLビュワーを開発。`cygnus_m_viewer_v6.html` が現在の確定版。キー配置・pos番号・レイヤー構成はこのViewerで確認できる。

### フェーズ3: Typing Trainer 開発（ChatGPT → Claude）

Viewer をベースに、Cygnus-M専用のタイピング練習ブラウザを開発。
最初のベース版は ChatGPT が作成（`cygnus_m_typing_trainer.html` 初期版）。
その後、初期版のUIとデザインを正として引き継ぎ、Claude が機能拡張を担当。

**引き継ぎ時の判断:**
ChatGPTとの開発で拡張されたバージョンが存在したが、デザイン崩れ等の懸念から**初期版UIを正として再構築する方針**を採用した。今後の修正・拡張もこの方針を維持する。

## 変更してはいけないもの

- ダークテーマの基調（背景色・グラデーション）
- 練習パネルがキーボードの上に来るレイアウト順序
- キーボードビュワーの物理配置（pos番号・左右の並び・親指段）
- 黄色ハイライト（次のキー）と水色ハイライト（ホールドキー）の色と役割
- 統計表示（進捗 / ミス / 正確性 / WPM）の4項目構成
- `validate()` による51キー整合性チェックの維持

## 実装済み機能

### 動作確認済み

| 機能 | 概要 |
|---|---|
| ホーム列練習 | F/J周辺のホームポジションキーを反復出題 |
| Base全文字練習 | L0共通の英字・基本記号を出題 |
| L8記号数字練習 | 英数(LANG2)長押し中の記号・数字を出題。ホールドキーを水色でハイライト |
| L9ナビ/Fn練習 | RET長押し中の矢印・Fnキーを出題。ホールドキーを水色でハイライト |
| ミックス練習 | Base / L8 / L9 をランダムに混合出題 |
| 日本語ローマ字練習 | ひらがな・カタカナを固定文でローマ字入力練習。複数ローマ字候補・拗音・っ対応 |
| キーボードビュワー連動 | 練習中、次に押すキーがキーボードビュワー上でハイライトされる |
| 51キー整合性バリデーション | 起動時に全レイヤーのキー数とpos番号重複をチェック |

### 未実装（今後の予定）

| 機能 | 優先度 | 概要 |
|---|---|---|
| カスタムテキスト練習 | 高 | ユーザーが任意文章を貼り付けて練習 |
| ミスキー復習モード | 中 | ミスしたキーだけを再出題。ミス回数に応じて出題比率を調整 |
| 表示文/入力用テキスト分離モード | 中 | 漢字混じりの表示文と、ひらがなで書いた入力用テキストを分けて練習 |

## アーキテクチャ概要

### 主要な状態変数

| 変数 | 用途 |
|---|---|
| `activeLayer` | 現在表示中のレイヤーID（0〜10） |
| `activeMode` | 現在の練習モード（`home` / `base` / `symnum` / `nav` / `mixed` / `jp`） |
| `drill` | 英字・記号モード用の出題リスト（`makeTarget` オブジェクトの配列） |
| `drillPos` | drillの現在位置 |
| `jpDrill` | 日本語モード用のかなトークン列 |
| `jpDrillPos` | jpDrillの現在位置 |
| `jpInput` | 日本語モードで現在のかなに対して入力済みの文字列 |
| `missCount` / `keyStrokes` | 統計計算用カウンター |

### 主要な関数

| 関数 | 役割 |
|---|---|
| `startDrill()` | モードに応じてドリルを初期化。jpモードは `buildJpDrill()` に分岐 |
| `handlePracticeKey(event)` | キー入力の受け口。jpモードは `handleJpKey()` に分岐 |
| `handleJpKey(event)` | 日本語ローマ字モード専用のキー判定ロジック |
| `renderPracticePanel()` | 英字・記号モードのUI更新 |
| `renderJpPanel()` | 日本語モードのUI更新 |
| `render()` | キーボードビュワー全体の再描画 |
| `renderKeyboard()` | キーボード部分のみ再描画（jpモードのレイヤー更新に使用） |
| `validate()` | 起動時の51キー整合性チェック |
| `tokenizeKana(text)` | ひらがな文字列をかなトークン列に分解（拗音・っ処理含む） |
| `poolForMode(mode)` | モードに対応する出題プール（makeTargetの配列）を返す |

### makeTarget の構造

```javascript
makeTarget(
  display,      // 画面表示文字（例: "A", "し"）
  expected,     // 期待する入力（例: "a", "s"）
  layer,        // 対応するレイヤーID
  index,        // キーボード上のposインデックス
  hold,         // ホールドヒント文字列
  holdIndices,  // ホールドすべきキーのposインデックス配列
  aliases       // 代替入力候補配列（例: ["shi"]）
)
```

### 日本語ローマ字モードの判定ロジック

```
かなトークン1つに対して:
  1. 入力済み文字列(jpInput) + 今回のキーを tentative とする
  2. 全候補（primary + aliases）のいずれかに前方一致 → jpInput を更新して続行
  3. 全候補のいずれかと完全一致 → かな1文字完成、jpDrillPosを進める
  4. どの候補にも前方一致しない → ミス、jpInputをリセット
```

## 現在の制限事項（既知）

- **漢字未対応**: 日本語ローマ字モードはひらがな・カタカナのみ。漢字を含む文章は未対応
- **IME干渉**: 日本語モード使用時はIMEをOFFにする必要がある。ONの場合、`isComposing` フラグで弾くが環境によって挙動が異なる可能性がある
- **日本語モードのWPM**: `jpDrillPos`（かな数）をベースにWPMを算出しているため、英字モードとの比較には不向き
- **モバイル非対応**: 物理キーボード入力前提の設計。スマートフォン等のソフトキーボードでは正誤判定が正しく動作しない

## 更新ルール

1. keymap を変更した場合、Trainer のキー配置データ（`layers` 配列）も同期する
2. Viewer のキー配置を変更した場合、Trainer も同じ変更を反映する
3. 機能を追加した場合、「実装済み機能」テーブルを更新する
4. デザインを変更した場合、カラーパレット・ハイライト仕様テーブルを更新する
5. 既知の制限事項が解消された場合、「現在の制限事項」から削除する
